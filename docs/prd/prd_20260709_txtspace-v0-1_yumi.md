# TXTSpace v0.1 제품 요구사항 정의서(PRD)

나의 생각이 하나의 우주가 되다.

문서 버전: v1.2 / 대상 제품 버전: TXTSpace v0.1 / 작성일: 2026-07-09 / 작성자: SVIL 유미

> 상위 문서: **TXT 패밀리 마스터 — 목적과 공통 프로토콜**. 본 PRD는 마스터 §3 공통 Keyword/Context API 규격(schema v1.0)을 소비 측 관점에서 구현한다.
> 출처: SVIL Outline 위키 (2026-07-12 로컬 복제). TXTMyWorld의 바로 아래 레이어(보기)이자 형제 문서 — 참조용.

변경 이력:

* v1.0 (2026-07-08): 최초 작성 (MindBoard "생각 은하수" 분리)
* v1.1 (2026-07-08): 데이터 연동을 **로컬 read-only Keyword API(실시간)**로 확정. 개발 시점은 TXTDiary RC 완료 후 순차로 확정.
* v1.2 (2026-07-09, 유미): **2소스 → 3소스 통합**으로 확장. TXTAIMemory(AI 대화·실작업 맥락)를 세 번째 데이터 소스로 추가. TXT 패밀리 마스터 프로토콜 상속 명시. `source` 필드 기반 레이어 구분, `ai_id` 확장 필드 소비, TXTMyWorld(가칭)와의 관계 정의.

문서 목적: TXTSpace를 SVIL 개인 지능 생태계의 **통합 시각화 허브**로 정의한다. TXTDiary(기록)·TXTBrain(지식)·TXTAIMemory(AI 대화) 세 맥락화 소스를 하나의 우주 맵으로 시각화하고, 각 노드에서 원본 앱으로 딥링크한다.

## 1. 제품 비전과 철학

### 1.1 제품 정의

TXTSpace는 사용자의 생각(TXTDiary 키워드), 지식(TXTBrain 문서 키워드), 그리고 AI와의 대화·작업(TXTAIMemory 키워드)을 하나의 우주 맵으로 시각화하는 독립 구동형 앱이다. 별(키워드)의 크기는 빈도, 색은 감정, 별자리(연결선)는 맥락과 맥락의 관계를 나타내며, 별이 속한 **성단(레이어)**은 그 키워드가 어느 소스에서 왔는지(기록/지식/AI 대화)를 보여준다.

TXTDiary가 "기록", TXTBrain이 "지식", TXTAIMemory가 "AI와 함께한 작업"이라면, TXTSpace는 그 셋 사이의 **연결을 발견하는 공간**이다. MindBoard PRD의 Knowledge Bridge 개념은 TXTSpace가 계승한다.

### 1.2 왜 별도 앱인가

* 시각화는 세 맥락화 앱과 성격·기술·업데이트 주기가 다르다. 분리하면 각 앱이 가벼워지고, TXTSpace는 렌더링 품질에 집중할 수 있다.
* TXTSpace는 세 소스를 **동시에** 다뤄야 하므로, 어느 한 앱에 종속되면 생태계 확장이 어려워진다.
* 시각화를 원하지 않는 사용자는 각 맥락화 앱만 가볍게 쓸 수 있다.

### 1.3 보기 레이어라는 위치 (TXTMyWorld와의 경계)

TXT 패밀리에는 시각화 위에 한 겹이 더 있다. **TXTMyWorld(가칭)**는 세 소스를 가로질러 연관을 찾고 새 맥락을 생성하는 상위 레이어다. TXTSpace는 그 아래에서 **보기**를 담당한다.

* TXTSpace = 지도를 그린다. 무엇이 어디에 있고 어떻게 이어지는지 **보여준다.**
* TXTMyWorld = 그 지도 위에서 "여기서 무엇이 새로 태어나는가"를 **만든다.**
* 경계 원칙: TXTSpace는 read-only 소비·표시까지. 새 연결의 생성·저장은 하지 않는다.

### 1.4 핵심 가치

* Read-only by Design: 원본 데이터를 절대 건드리지 않는다. TXTSpace가 오류를 내도 일기·문서·AI 기억은 안전하다.
* Local-first: 키워드 데이터는 로컬에서만 읽고 처리한다. 외부 전송 없음.
* Accessible Visualization: 그래프 뷰와 동등한 **리스트 뷰**를 항상 제공한다. 시각화 앱이라도 저시력·스크린리더 사용자를 배제하지 않는다.
* Ecosystem Hub: 하나의 키워드에서 일기(TXTDiary)·자료(TXTBrain)·AI 기억(TXTAIMemory)으로 모두 이동할 수 있는 유일한 화면을 지향한다.

## 2. 생태계 내 위치

### 2.1 데이터 흐름 (3소스)

```
TXTDiary ────(Keyword API: localhost read-only, source=txtdiary)────┐
TXTBrain ────(Keyword API: localhost read-only, source=txtbrain)────┤──▶ TXTSpace (보기)
TXTAIMemory ─(Keyword API: localhost read-only, source=txtaimemory)─┘        │
                                                                             ▼
                        노드 클릭 ──▶ txtdiary://search?keyword=...   (일기 보기)
                        노드 클릭 ──▶ txtbrain://search?keyword=...   (자료 보기, v0.5+)
                        노드 클릭 ──▶ txtaimemory://recall?keyword=... (AI 기억 회상, v0.5+)

TXTSpace ──(read-only 통합 뷰 제공)──▶ TXTMyWorld (잇기·만들기, 상위 레이어)
```

### 2.2 버전별 데이터 소스

| 버전  | TXTDiary | TXTBrain | TXTAIMemory | 통합 뷰 |
|-----|----------|----------|-------------|------|
| v0.1 | **Keyword API 실시간 조회** + JSON 파일 폴백 | —        | —           | 단일 소스 |
| v0.5 | 동일       | Keyword API 읽기 | **Keyword API 읽기** | 소스별 레이어 토글 |
| v1.0 | API 변경 이벤트 push(SSE, TXTDiary v1.5 연동) | 동일       | 동일          | 공통 키워드 하이라이트, 교차 연결선(기록↔지식↔AI대화) |

### 2.3 공통 Keyword API 스키마 (마스터 §3 상속)

TXT 패밀리 마스터 §3.2의 공통 응답 스키마(schema v1.0)를 그대로 소비한다. 스키마는 패밀리 전체가 공동 소유하며, `schema_version` 필드로 호환성을 관리한다. 알 수 없는 상위 버전 스키마를 만나면 오류 대신 "TXTSpace 업데이트 필요" 안내를 표시한다.

* 세 소스 모두 동일한 엔드포인트(`/health`, `/keywords`, `/keywords/{normalized_text}/cooccurrence`)와 동일한 응답 형태를 제공하므로, TXTSpace는 **소스별로 같은 클라이언트 코드를 재사용**한다. 구분은 오직 `source` 필드와 포트로 한다.
* `source` 필드로 노드의 레이어·색 계열을 나눈다. (기록/지식/AI 대화)
* TXTAIMemory 응답의 확장 필드 `ai_id`는 알면 활용(AI별 세부 필터), 모르면 무시(graceful degradation)한다.
* 각 소스와 최초 실행 시 개별 페어링을 수행한다. 각 앱 승인 다이얼로그에서 사용자가 동의하면 페어링 토큰을 받아 OS 보안 저장소에 저장한다.
* 어떤 소스가 실행 중이 아니면 그 소스만 마지막 캐시/JSON 폴백으로 표시하고 "오프라인 데이터" 배지를 붙인다. 나머지 소스는 정상 표시한다.

## 4. 사용자 시나리오

1. 사용자가 TXTSpace를 처음 열면 TXTDiary와 페어링한다(TXTDiary 승인 다이얼로그에서 1회 동의).
2. 이후 TXTSpace를 열 때마다 Keyword API에서 최신 키워드가 실시간으로 로드되어 우주가 표시된다.
3. 큰 별(자주 등장한 키워드)과 색(감정)으로 최근 생각의 지형을 한눈에 본다.
4. "양자역학" 별을 클릭하면 TXTDiary가 열리며 관련 일기가 검색된다.
5. 기간 필터를 "지난달 → 이번 달"로 바꿔 관심사의 이동을 확인한다.
6. (v0.5+) 같은 화면에서 소스 레이어를 켜면, 같은 "양자역학" 별에 TXTBrain의 관련 문서와 TXTAIMemory의 관련 AI 대화가 다른 색 성단으로 함께 뜬다. 하나의 키워드가 기록·지식·AI 작업에 어떻게 걸쳐 있는지 한눈에 보인다.

### 4.1 주요 사용자 스토리

* 성찰형 기록자로서, 내 키워드들이 우주처럼 펼쳐진 걸 보고 싶다. 내 생각의 지형을 직관적으로 파악하기 위해서다.
* 성찰형 기록자로서, 특정 키워드를 클릭해 관련 일기로 바로 이동하고 싶다. 시각화가 탐색의 출발점이 되기를 원한다.
* 창작자/연구자로서, 하나의 개념이 내 기록·자료·AI와의 작업에 각각 어떻게 나타나는지 한 화면에서 비교하고 싶다. 흩어진 맥락이 실제로 이어져 있음을 보고 싶다. (v0.5+)
* 저시력 사용자로서, 그래프 대신 빈도순 키워드 리스트로 같은 정보를 탐색하고 싶다. 시각화 앱에서도 배제되고 싶지 않다.
* 테크니컬 유저로서, TXTSpace가 내 원본을 읽거나 수정하지 않는다는 걸 확인하고 싶다. 데이터 안전을 신뢰하기 위해서다.

## 5. 핵심 기능 상세 명세

### 5.1 데이터 로드

| 항목  | 요구사항 |
|-----|------|
| 페어링 | 소스별 최초 연결 시 각 앱 승인 다이얼로그 기반 페어링. 토큰은 OS 보안 저장소에 저장 |
| 실시간 조회 | 시작 시 각 활성 소스에 GET /keywords 호출, 이후 주기적 polling(기본 60초, 조절 가능)으로 갱신 |
| 소스 병합 | 여러 소스 응답을 `source` 필드 기준으로 병합. 같은 normalized_text는 소스별로 별도 노드로 두되, 교차 연결선으로 이어 표시(v0.5+) |
| 연결 실패 폴백 | 특정 소스 미실행/미응답 시 그 소스만 마지막 캐시 또는 JSON 폴백 로드 + "오프라인 데이터" 배지. 다른 소스는 정상 표시 |
| 수동 파일 로드 | 파일 선택 대화상자로 export JSON 로드 (폴백/이동용) |
| 스키마 검증 | schema_version 확인. 미래 버전이면 업데이트 안내, 손상 응답/파일이면 명확한 오류 표시 |
| 프라이버시 | 본문 필드가 응답/파일에 포함되어 있어도 무시(파싱하지 않음) |

### 5.2 워드클라우드 뷰 (기본)

| 요소  | 의미  | 구현  |
|-----|-----|-----|
| 글자 크기 | 키워드 빈도 | frequency 정규화 |
| 색상  | 감정 성향 | avg_emotion_score → 긍정/중립/부정 팔레트 (고대비 팔레트 별도 제공) |
| 성단(레이어) | 소스 종류 | source(txtdiary/txtbrain/txtaimemory) → 색 계열/그룹 구분 (v0.5+) |
| 클릭  | 원본 앱 열기 | source별 딥링크 (txtdiary://search / txtbrain://search / txtaimemory://recall) |
| 호버/포커스 | 상세 툴팁 | 빈도, 감정 점수, 첫/마지막 등장일, 동시 출현 상위 3개, 소스, (AI 대화면) ai_id |

### 5.3 2D 그래프 맵 (Should)

| 요소  | 의미  | 구현  |
|-----|-----|-----|
| 노드  | 키워드 | 크기=빈도, 색=감정, 테두리/그룹=소스 |
| 연결선 | 동시 출현 | cooccurrence count 기준 상위 연결만 표시(기본 노드당 최대 5개) |
| 교차 연결선 | 소스 간 공통 키워드 | 같은 normalized_text가 여러 소스에 있으면 소스 간 브리지 선으로 연결(v0.5+) |
| 레이아웃 | 관계 기반 배치 | force-directed layout, 노드 수 상한(기본 100개, 조절 가능) |
| 조작  | 팬/줌/노드 드래그 | 마우스 + 키보드(방향키 노드 이동, +/- 줌) |

### 5.4 리스트 뷰 (접근성 필수)

* 그래프의 모든 정보를 표 형태로 제공: 키워드, 빈도, 감정, 기간, 동시 출현 키워드, **소스**.
* 정렬(빈도/감정/최근성/소스), 검색 필터 지원.
* 스크린리더가 각 행을 완전한 문장으로 읽을 수 있도록 접근 가능한 이름 제공. 소스도 문장에 포함("…, AI 대화 맥락에서 12회 등장").
* 리스트 뷰에서도 Enter로 딥링크 실행 가능. 그래프 뷰와 기능 동등성(feature parity)을 유지한다.

## 6. 시스템 아키텍처

| 영역  | 기술  | 비고  |
|-----|-----|-----|
| 프레임워크 | Tauri + React | TXT 패밀리 스택 통일, 코드/컴포넌트 재사용 |
| 시각화 | D3.js (워드클라우드/force graph) | v1.0의 3D는 Three.js 검토 |
| 데이터 | Keyword API HTTP 클라이언트(127.0.0.1) × 소스 수 + JSON 파싱, 인메모리 병합 | 자체 DB 없음(설정 파일 + 폴백 캐시만 보유) |
| 딥링크 발신 | OS shell open으로 txtdiary:// / txtbrain:// / txtaimemory:// 호출 | 대상 앱 미설치 시 안내 표시 |
| 설정  | 로컬 JSON | 소스별 페어링, 뷰 옵션, 접근성 프로필 |

* **자체 DB를 갖지 않는 것이 원칙이다.** TXTSpace의 상태는 언제든 API 재조회로 복원 가능해야 한다. 폴백 캐시는 마지막 API 응답의 스냅샷일 뿐이다.
* 소스가 늘어도 클라이언트 코드는 하나. 공통 스키마 덕분에 소스 추가는 "새 포트/페어링 등록"에 가깝다.

## 8. 접근성 요구사항

* 리스트 뷰는 그래프 뷰와 항상 기능 동등성을 유지한다. (핵심 원칙)
* 키보드만으로 뷰 전환, 노드 탐색, 필터, 소스 토글, 딥링크 실행이 가능해야 한다.
* 고대비 모드: 감정 색상 팔레트를 고대비 버전으로 교체 + 색 이외의 구분 수단(도형/패턴) 병행. **소스 구분도 색만이 아니라 도형/그룹으로 병행**한다.
* 최소 폰트 16px, 클릭 타겟 50px. (SVIL 접근성 표준)
* 워드클라우드 최소 글자 크기 하한 설정 제공(빈도가 낮아도 읽을 수 있게).
* NVDA/Narrator로 리스트 뷰 전체 흐름 테스트.
* 400% 확대에서 리스트 뷰와 필터가 깨지지 않아야 한다.

## 9. 테스트 및 Acceptance Criteria

| 기능  | Given | When | Then |
|-----|-------|------|------|
| 페어링 | TXTDiary가 실행 중이다 | 최초 연결을 시도한다 | TXTDiary 승인 후 토큰이 보안 저장소에 저장된다 |
| API 조회 | 페어링이 완료됐다 | 앱을 실행한다 | /keywords 응답으로 워드클라우드가 표시된다 |
| 다중 소스 병합 | Diary·Brain·AIMemory가 모두 페어링됐다(v0.5) | 앱을 실행한다 | 세 소스 키워드가 source별 레이어로 병합 표시된다 |
| 소스 폴백 격리 | Diary만 실행 중이고 Brain은 꺼져 있다(v0.5) | 앱을 실행한다 | Diary는 실시간, Brain만 오프라인 배지로 표시되고 앱은 정상 동작한다 |
| 스키마 방어 | schema_version이 2.0인 응답/파일이 있다 | 로드한다 | 오류 대신 업데이트 안내가 표시된다 |
| 확장 필드 무시 | 응답에 모르는 필드(ai_id)가 있다 | 파싱한다 | 오류 없이 무시하고 코어 필드로 렌더한다 |
| 딥링크(Diary) | TXTDiary가 설치되어 있다 | "양자역학" 노드를 클릭한다 | TXTDiary 검색 결과가 keyword=양자역학으로 열린다 |
| 딥링크(AIMemory) | TXTAIMemory가 설치되어 있다(v0.5) | AI 대화 소스 노드를 클릭한다 | txtaimemory://recall?keyword=…로 회상 화면이 열린다 |
| 미설치 안내 | 대상 앱이 설치되어 있지 않다 | 노드를 클릭한다 | 설치 안내가 표시되고 앱은 크래시하지 않는다 |
| 기간 필터 | 여러 기간의 키워드가 있다 | 필터를 "이번 달"로 바꾼다 | 해당 기간 키워드만 재계산되어 표시된다 |
| 소스 토글 | 세 소스가 로드되어 있다(v0.5) | Brain 레이어를 끈다 | Brain 노드만 사라지고 나머지는 유지된다 |
| 리스트 뷰 동등성 | 그래프에 노드 N개가 있다 | 리스트 뷰로 전환한다 | 동일한 N개 키워드와 정보(소스 포함)가 표에 표시된다 |
| 키보드 탐색 | 마우스를 사용하지 않는다 | Tab/방향키/Enter로 조작한다 | 노드 탐색과 딥링크 실행이 가능하다 |
| 읽기 전용 | 데이터가 로드되어 있다 | 어떤 조작을 해도 | 어떤 소스의 DB/API 데이터도 변경되지 않는다(쓰기 요청을 보내지 않음) |
