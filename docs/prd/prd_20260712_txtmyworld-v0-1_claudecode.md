## 1. 제품 비전과 철학

### 1.1 제품 정의

TXTMyWorld는 사용자의 **기억(TXTDiary)·지식(TXTBrain)·AI 작업(TXTAIMemory)** 에서 나온 키워드들을 한데 모아, 기간·빈도·벡터로 교차 조합해 **서로 다른 맥락 사이에 숨어 있던 연결과 아직 이름 붙지 않은 주제를 발견**하는 앱이다. 발견에 그치지 않고, 그 주제를 **카드로 명명·저장**하고 필요하면 생태계로 되돌려 보낸다.

사람의 뇌가 자는 동안 학교 일·집안일·친구 일을 묶어 새로운 통찰을 만들 듯, TXTMyWorld는 흩어진 개인 맥락을 가로질러 **"여기서 무엇이 새로 태어나는가"** 에 답한다.

### 1.2 왜 이 앱이 생태계의 핵심인가

맥락화 3축은 각자 자기 도메인을 잘 정리하고, TXTSpace는 그것을 한 지도로 보여준다. 하지만 **보는 것과 만드는 것은 다르다.** 지도를 아무리 잘 그려도, 서로 다른 성단 사이를 잇는 새 별자리를 그리는 일은 별도의 의도적 작업이다. TXTMyWorld는 바로 그 일 — 소스를 가로지르는 **연결의 생성** — 을 전담한다. 생태계가 존재하는 최종 이유(성장·통찰)가 실현되는 지점이다.

### 1.3 핵심 가치

* **Cross-source by Design**: 한 소스 안이 아니라 **소스와 소스 사이**를 본다. 단일 소스 분석은 각 맥락화 앱의 몫.
* **재정리가 곧 발견**: 저장이 아니라 묶고·요약하고·연결하고·버리는 것. 원장→통합→망각 루프(마스터 §4)를 발견 로직에 녹인다.
* **Local-first & Token is Time**: 벡터·조합 연산은 로컬 우선. 클라우드는 사용자가 명시적으로 원할 때만.
* **Read the sources, own only what you create**: 세 소스는 절대 건드리지 않는다. TXTMyWorld는 자기가 **만든 것(주제 카드)만** 소유한다.
* **Accessible Discovery**: 시각 그래프와 동등한 리스트/설명 뷰를 항상 제공. 발견의 근거를 문장으로 읽을 수 있어야 한다.

### 1.4 한 줄 슬로건 (후보)

* "잇고, 만들다." / "아직 이름 없는 주제를 찾아서." / "내 맥락들이 만나는 곳."

## 2. 생태계 내 위치

### 2.1 역할 경계 (마스터 §2.1 준수)

```
[맥락화 3축 · read-only 제공]        [시각화 · 보기]         [연결 · 만들기]
TXTDiary  (개인 기억) ─┐
TXTBrain  (문서 지식) ─┼─ Keyword/Context API ─▶ TXTSpace ─▶ TXTMyWorld
TXTAIMemory (AI 대화) ─┘        (schema v1.0)      (지도)     (주제 생성 · 환류)
                                                                  │
                                            생성한 주제 ──(동의 시 MCP 쓰기)──▶ TXTAIMemory
```

* **TXTSpace = 보기.** 세 소스를 지도에 얹어 보여준다. 새 연결을 생성·저장하지 않는다.
* **TXTMyWorld = 잇기·만들기.** 세 소스를 가로질러 연관을 찾고, 새 주제를 생성·저장하며, 동의 시 생태계로 환류한다.
* **맥락화 3앱 = 원본 소유자.** 자기 키워드를 read-only로 제공할 뿐, 서로/상위 레이어를 읽지 않는다.
* **혼동 방지:** TXTMyWorld가 지도를 "예쁘게 그리는" 데 힘을 쏟으면 TXTSpace와 겹친다. 본 앱의 존재 이유는 **생성**이다. 시각화는 발견을 돕는 보조 수단일 뿐.

### 2.2 데이터 소스 & 통합 조회 (마스터 §3 상속)

세 소스 모두 동일한 로컬 read-only Keyword/Context API(schema v1.0)를 노출하므로, TXTMyWorld는 **소스별로 같은 클라이언트 코드를 재사용**해 통합 조회한다.

| 항목 | 내용 |
|-----|-----|
| 엔드포인트 | 각 소스의 `/health`, `/keywords`, `/keywords/{normalized_text}/cooccurrence` |
| 소스 구분 | 응답의 `source` 필드(txtdiary/txtbrain/txtaimemory). AIMemory는 `ai_id` 확장 필드로 AI별 세부 구분 |
| 인증 | 소스별 최초 페어링(각 앱 승인 다이얼로그) → 토큰 OS 보안 저장소 저장 |
| 프라이버시 | 응답에 **본문 없음**(키워드·빈도·감정·동시출현·first/last_seen만). 벡터가 필요하면 §8 참조 |
| 폴백 | 특정 소스 미실행 시 그 소스만 마지막 캐시로 표시 + "오프라인 데이터" 배지. 나머지 소스로 발견 계속 |
| 스키마 방어 | schema_version 확인. 상위 버전이면 오류 대신 "업데이트 필요" 안내, 모르는 필드는 무시 |

> **규격 결정 (D1, 확정):** 현재 공통 API 응답은 키워드 메타데이터 중심이며 임베딩 벡터를 포함하지 않는다. 벡터가 **핵심 축**이므로 **이중 전략**으로 확정한다 — **(A) 소스측 임베딩 공유**(RC 목표): 공통 API를 schema v1.1로 확장해 각 소스가 자기 본문 맥락으로 계산한 임베딩 **벡터만** 선택 공유(본문은 여전히 미전송). **(B) 로컬 자체 임베딩**(v0.1 부트스트랩/폴백): 소스가 벡터를 주지 않으면 TXTMyWorld가 키워드 텍스트(+동시출현 이웃)로 로컬 임베딩. 소비 측은 벡터가 있으면 (A), 없으면 (B)로 graceful하게 동작한다. 상세 §3.4.

## 3. 핵심 개념 — 발견 엔진 (기간 · 빈도 · 벡터 조합)

TXTMyWorld의 심장. 세 축을 각각, 그리고 **조합**해 "새 주제 후보"를 만든다.

### 3.1 세 가지 조합 축

| 축 | 무엇을 보는가 | 발견 예시 |
|---|-----------|--------|
| **기간(temporal)** | 같은 시간대에 여러 소스에서 함께 떠오른 키워드, 관심의 이동 | "지난달 Diary의 '번아웃'과 AIMemory의 '일정 압박'이 같은 주에 급증 → 겹치는 시기" |
| **빈도(frequency)** | 소스 간 빈도 상관, 급상승/급하강, 소스별 불균형 | "Brain엔 '양자측정'이 많은데 Diary엔 없음 → 아는데 안 쓴 생각" |
| **벡터(semantic)** | 표현은 다르지만 의미가 가까운 키워드의 교차 | "Diary '관측자' ↔ Brain 'measurement problem' ↔ AIMemory 'observer effect' → 같은 개념의 세 얼굴" |

### 3.2 조합에서 나오는 "발견 유형"

* **브리지(bridge):** 서로 다른 소스의 키워드가 벡터·동시출현으로 강하게 이어짐 → 이미 연결됐으나 사용자가 의식 못 한 관계.
* **갭(gap):** 한 소스엔 강한데 다른 소스엔 비어 있음 → "지식은 있는데 경험이 없다" / "경험했는데 정리 안 했다".
* **이머전트 클러스터(emergent cluster):** 세 소스에 흩어진 키워드들이 하나의 의미 덩어리를 이루는데 **아직 이름이 없음** → 새 주제 후보. **본 앱의 대표 산출물.**
* **드리프트(drift):** 같은 키워드의 감정/맥락이 기간에 따라 이동 → 생각의 변화 궤적.

### 3.3 발견 → 생성 흐름

```
1. 수집   세 소스 Keyword/Context API 통합 조회 (+ 로컬 임베딩 생성/캐시)
2. 조합   기간·빈도·벡터 축으로 후보 계산 (브리지/갭/클러스터/드리프트)
3. 제시   [하이브리드] 추천 피드(자동) + 탐색 캔버스(수동 필터·조합)
4. 명명   사용자가 후보를 '주제 카드'로 채택하고 이름을 붙임 (또는 AI 제안 라벨 수락)
5. 저장   주제 카드를 자체 DB에 저장 (근거·구성 키워드·소스 딥링크 포함)
6. 환류   [동의 시] 주제 카드를 TXTAIMemory에 MCP 쓰기로 통합 기억으로 등록
```

### 3.4 벡터·의미 검색 엔진 (핵심 · RC 최종 지향)

> 이 절이 TXTMyWorld의 심장이다. 기간·빈도는 **표면**(표기가 같은 키워드)을 잇지만, 표현이 다르면 놓친다. **벡터(의미)만이 "관측자 = observer effect = 측정 문제"처럼 표기가 달라도 같은 개념을 잇는다.** 그래서 벡터는 v0.1의 임시 기능이 아니라 **처음부터 핵심 축으로 설계하고, RC(v1.0)까지 최종에 가깝게 완성**한다. 아래 설계는 v0.1에서 축소 구현하더라도 **인터페이스·데이터 모델은 RC 형태를 미리 반영**해, 나중에 갈아엎지 않도록 한다.

#### 3.4.1 임베딩 소스 — 이중 전략 (D1 확정)

| 전략 | 방식 | 프라이버시 | 위치 |
|-----|-----|--------|-----|
| **(A) 소스측 임베딩 공유** | 각 소스(Diary/Brain/AIMemory)가 **자기 본문 맥락**으로 키워드를 임베딩하고, 공통 API(schema v1.1 확장)로 **벡터만** 선택 공유. 본문은 여전히 미전송 | 최상 — 풍부한 의미(본문 반영)이면서 원문 불출 | RC 목표 |
| **(B) 로컬 자체 임베딩** | 소스가 벡터를 안 주면 TXTMyWorld가 **키워드 텍스트(+동시출현 이웃 몇 개를 문맥으로)** 를 로컬 임베딩 | 상 — 전부 로컬, 외부 전송 없음 | v0.1 부트스트랩/폴백 |

* 소비 로직은 **graceful**: 소스별로 벡터가 있으면 (A), 없으면 (B)로 자동 폴백. 세 소스가 서로 다른 상태여도 각각 최선으로 처리.
* **혼용 정합성:** (A)와 (B), 또는 소스마다 다른 임베딩 모델이 섞이면 벡터 공간이 어긋난다. 대응 — ①벡터마다 `model`·`dim` 태깅, ②**같은 모델끼리만 직접 비교**, ③이종 모델 간 비교가 필요하면 로컬 재임베딩으로 한 공간에 정렬(재색인). v0.1은 단순화를 위해 **로컬 단일 모델로 전 소스 재임베딩(B)** 을 기본 경로로 삼고, (A)는 v0.5+에서 벡터가 준비되는 소스부터 점진 도입.

#### 3.4.2 임베딩 모델 (D2 — 기본 확정)

| 항목 | 결정 |
|-----|-----|
| 요구 | **한국어+영어 다국어** 의미 정합이 필수(사용자 콘텐츠가 한/영 혼용). 짧은 키워드+짧은 문맥에 강해야 함 |
| 기본 모델 | **bge-m3** (다국어, 1024-dim) — 로컬(Ollama/llama.cpp) 구동. 대안: multilingual-e5-large(1024-dim) |
| 유사도 | 코사인(벡터 L2 정규화 후 내적) |
| 버전 관리 | 벡터에 `model`·`dim`·`normalized` 태깅. **모델 교체 시 전체 재색인**(마이그레이션 잡). 사용자 하드웨어(RTX 5060Ti 16GB) 기준 배치 임베딩 가능 |

#### 3.4.3 벡터 스토어 & 인덱스

| 항목 | 결정 |
|-----|-----|
| 저장 | 앱 SQLite 내 **sqlite-vec(vec0 가상 테이블)** — 로컬 KNN. 별도 서버 없음(local-first) |
| 대규모 | 벡터 수가 커지면 **HNSW(ANN)** 인덱스로 승격(근사 최근접, 상한·efSearch 튜닝) |
| 갱신 | **증분 upsert** — 변경/신규 키워드만 재임베딩·재색인. 백그라운드 처리로 UI 비블로킹 |
| 메타 | 각 벡터 레코드: `source`, `normalized_text`, `model`, `dim`, `created_at`. 소스·모델별 필터 검색 지원 |

#### 3.4.4 벡터가 여는 의미 연산 (핵심 기능군)

| 연산 | 설명 | 발견 유형 매핑 |
|-----|-----|-----------|
| **교차소스 개념 매칭** | 표기가 달라도 의미가 가까운 키워드를 소스 간에 이음 | 브리지 |
| **의미 클러스터링** | 세 소스에 흩어진 벡터를 밀도 기반(HDBSCAN)·커뮤니티 검출로 묶어 **아직 이름 없는 주제 덩어리** 추출 | 이머전트 클러스터(대표 산출물) |
| **의미 갭 탐지** | 한 소스엔 조밀한 의미 영역이 다른 소스엔 비어 있음 | 갭 |
| **의미 기반 질의** | 자연어/예시 키워드 → 임베딩 → 최근접 이웃 검색("이런 느낌의 주제 찾아줘") | 탐색형 발견 |
| **의미 드리프트** | 같은 개념 클러스터의 중심 벡터가 기간에 따라 이동하는 궤적 | 드리프트 |

#### 3.4.5 3축 융합 — 벡터를 뼈대로, 기간·빈도로 가중 (결정적)

* **뼈대:** 벡터 최근접/클러스터가 후보의 골격을 만든다(의미 이웃).
* **가중·필터:** 기간(같은 시기 동시 등장?)과 빈도(양쪽 다 유의미한 양?)가 후보의 **신뢰도**를 조정한다.
* **스코어:** `score = w_s·semantic_sim + w_t·temporal_overlap + w_f·frequency_signal` (가중치 조절 가능, 기본값은 개발 중 튜닝). 벡터 유사도가 낮으면 후보에서 제외 → **우연한 표면 일치로 인한 오탐 감소**.
* **재현성:** 임베딩·인덱스 상태가 고정되면 동일 입력에 동일 후보. 근거(유사도·기간·빈도 수치)를 그대로 제시(§6.2 투명성).

#### 3.4.6 성능·확장 (RC 기준)

* ANN 상한(후보 K, efSearch), 클러스터 최소 크기·밀도 임계값으로 폭주 방지.
* 배치 임베딩 + 증분 재색인 + 결과 캐시. 저사양 모드(임베딩 지연·KNN 상한 축소) 옵션.
* 목표: 3소스 합산 수만 키워드 규모에서 KNN 질의 체감 지연 없이 동작(구체 수치 §10).

#### 3.4.7 프라이버시 (벡터 특화)

* 벡터는 **로컬에만 저장**하고 외부로 전송하지 않는다.
* 소스측 공유(A)도 **벡터만** 공유하며 본문은 불출(마스터 §3 본문 미포함 원칙과 정합).
* 클라우드(DeepSeek) 합성은 **벡터가 아니라** 확정된 주제의 라벨/설명 텍스트에만, 동의 후 사용. 임베딩 원본은 클라우드로 보내지 않는다.

## 5. 사용자 시나리오

1. TXTMyWorld를 처음 열면 세 소스와 각각 페어링한다(실행 중인 소스만, 나머지는 나중에).
2. 홈에 **추천 피드**가 뜬다: "이번 주 새로 이어진 것 3건" — 브리지·갭·이머전트 클러스터 카드 후보.
3. "'관측자'(일기) ↔ 'measurement problem'(문서) ↔ 'observer effect'(AI 대화)가 한 덩어리로 보입니다"라는 클러스터 후보를 연다.
4. 근거 패널에서 어느 소스·어느 기간·어떤 유사도로 묶였는지 문장과 표로 확인한다.
5. 채택 후 **주제 카드**로 이름을 붙인다 — 예: "관측 문제: 나의 세 갈래". (AI 제안 라벨을 수락해도 됨)
6. 각 구성 키워드를 클릭하면 소스 딥링크로 원본 앱(일기 검색/문서/AI 회상)이 열린다.
7. "이 주제를 AI 기억에 남기기"를 켜면 동의 후 TXTAIMemory에 통합 기억으로 환류된다.
8. (탐색형) 다음엔 직접 "최근 30일 × Brain+AIMemory × 벡터 유사"를 필터로 걸어 원하는 교차를 판다.

### 5.1 주요 사용자 스토리

* 창작자/연구자로서, 내 기록·자료·AI 작업에 흩어진 조각이 하나의 주제로 묶이는 순간을 잡고 싶다. 창작·연구의 씨앗을 얻기 위해서다.
* 성찰형 기록자로서, "아는데 안 쓴 생각"(갭)을 알려주면 좋겠다. 나를 더 이해하기 위해서다.
* 테크니컬 유저로서, 발견이 어떤 근거로 나왔는지 투명하게 보고 싶다. 우연한 상관을 통찰로 오인하지 않기 위해서다.
* 저시력 사용자로서, 그래프 없이도 발견 목록과 근거를 문장으로 읽고 채택하고 싶다.
* 프라이버시 우선 사용자로서, 외부 전송 없이 로컬만으로 발견이 돌아가길 원한다.

## 6. 핵심 기능 상세 명세

### 6.1 통합 조회 & 로컬 임베딩

| 항목 | 요구사항 |
|-----|------|
| 페어링 | 소스별 최초 연결 시 승인 다이얼로그. 토큰 OS 보안 저장소 저장 |
| 조회 | 활성 소스에 GET /keywords 호출, 주기적 갱신(기본 수동/온디맨드 + 옵션 polling) |
| 병합 | `source` 기준 병합. 같은 normalized_text는 소스별 별도 항목으로 두되 교차 후보 계산에 사용 |
| 임베딩 | 키워드 텍스트로 로컬 임베딩 생성 후 캐시(변경분만 재계산). 본문은 다루지 않음 |
| 폴백 | 소스 미실행 시 격리 처리(그 소스만 오프라인 배지), 앱은 정상 동작 |

### 6.2 발견 엔진 (§3 구현)

* 기간·빈도 조합은 결정적(deterministic) 계산 — 재현 가능하고 근거를 수치로 제시.
* 벡터 유사 클러스터는 로컬 임베딩 + cosine similarity + 임계값/상한(성능 보호).
* 각 후보는 **발견 유형(브리지/갭/클러스터/드리프트)**, **구성 키워드+소스**, **근거(기간·빈도·유사도 수치)**, **신뢰도 힌트**를 갖는다.
* 우연 상관 방지: 최소 빈도/최소 소스 수/기간 겹침 임계값을 두고, 근거가 약하면 "약한 신호"로 라벨.

### 6.3 제시 — 하이브리드 (추천 + 탐색)

| 모드 | 동작 |
|-----|-----|
| 추천 피드 | 앱이 최신 후보를 자동 계산해 카드 후보 목록으로 제시. "새로 이어진 것 / 갭 / 떠오르는 클러스터" 섹션 |
| 탐색 캔버스 | 사용자가 소스·기간·축(기간/빈도/벡터)·최소 임계값을 필터로 걸어 직접 조합 |
| 관계 뷰(Should) | 후보 구성 키워드를 노드로, 근거를 엣지로 보여주는 간단 2D 뷰. 리스트 뷰와 동등성 유지 |

### 6.4 생성 — 주제 카드

* 후보를 **채택**하면 주제 카드가 자체 DB에 생성된다.
* 카드 필드: 이름(사용자/AI), 발견 유형, 구성 키워드 목록(text·source·weight), 근거 스냅샷(기간·빈도·유사도), 생성일, 상태(초안/확정/보관), 메모, 소스 딥링크.
* 카드는 **TXTMyWorld가 소유**하는 유일한 쓰기 데이터. 세 소스 원본과 무관하며, 소스가 바뀌면 근거 스냅샷은 "그때의 관찰"로 보존된다.
* 편집·삭제(soft delete) 가능. Markdown 내보내기(Could).

### 6.5 환류 — TXTAIMemory 연동 (동의형)

* 사용자가 명시적으로 켰을 때만, 주제 카드를 TXTAIMemory의 **MCP 쓰기 인터페이스**로 통합 기억(consolidated)으로 등록한다.
* 전송 범위는 메타데이터 중심(주제명·구성 키워드·근거 요약). 소스 본문은 전송하지 않는다.
* TXTAIMemory 미설치/미실행이면 이 기능은 비활성(회색 처리 + 안내), 앱은 정상 동작.
* 환류 이력은 카드에 기록(언제·무엇을 보냈는지)해 투명성을 보장한다.

### 6.6 합성 — 로컬 우선 + 동의형 클라우드

| 작업 | 기본(로컬) | 동의 시(클라우드) |
|-----|---------|--------------|
| 주제 라벨 제안 | 구성 키워드 기반 로컬 규칙/소형 모델 | DeepSeek 고품질 라벨 후보 |
| 주제 설명 문장 | 근거를 템플릿으로 서술 | DeepSeek 자연어 설명 |
| 트리거 | 항상 사용 가능 | 사용자가 버튼으로 요청할 때만, 1회 동의 확인 |

* 클라우드 호출 전 전송 데이터(키워드·근거 요약, 본문 없음)를 사용자에게 명시하고 동의를 받는다.
* 자동 과금 방지: 클라우드는 기본 OFF, 사용자 API Key 직접 조달(패밀리 정책 상속).

## 7. 접근성 요구사항

* **리스트/설명 뷰 동등성(핵심 원칙):** 관계 뷰의 모든 발견은 리스트+근거 문장으로 동일하게 제공. 스크린리더가 "무엇이 왜 이어졌는지"를 완전한 문장으로 읽을 수 있어야 한다.
* 키보드만으로 피드 탐색, 필터, 카드 채택·명명·저장, 딥링크 실행, 환류 토글이 가능.
* 고대비 다크 기본. 발견 유형은 색만이 아니라 텍스트 라벨·도형으로 병행 구분.
* 최소 폰트 16px, 클릭 타겟 50px. 400% 확대에서 리스트·필터 유지. 애니메이션 줄이기 옵션.
* 발견 근거 수치에는 항상 텍스트 설명을 병기(색/그래프만으로 전달 금지).

## 8. 데이터 설계 (자체 DB)

로컬 저장. 세 소스 원본은 소유하지 않으며, 아래는 **TXTMyWorld가 생성·소유**하는 데이터다.

| 테이블 | 목적 | 주요 필드 |
|-----|-----|-------|
| sources | 페어링된 소스 | id, source(txtdiary/txtbrain/txtaimemory), base_url, pairing_token_hash, last_synced_at |
| keyword_cache | 소스 조회 스냅샷 | id, source, normalized_text, text, category, frequency, avg_emotion_score, first_seen, last_seen, fetched_at |
| embeddings | 키워드 임베딩(소스별) | id, source, normalized_text, origin(local/shared), model, dim, normalized(bool), vector_blob, created_at |
| vec_index | 벡터 KNN 인덱스 | sqlite-vec(vec0) 가상 테이블 — embedding_id ↔ vector 매핑. 규모 확장 시 HNSW 파라미터(efSearch, M) 메타 포함 |
| clusters | 의미 클러스터 스냅샷 | id, method(hdbscan 등), member_embedding_ids_json, centroid_blob, period, created_at |
| discoveries | 발견 후보 | id, type(bridge/gap/cluster/drift), member_keys_json, evidence_json(semantic_sim·기간·빈도), score, status(new/dismissed/adopted), created_at |
| topic_cards | 생성된 주제 카드 | id, name, label_source(user/ai), discovery_id, member_keys_json, evidence_snapshot_json, note, status(draft/confirmed/archived), created_at, updated_at, deleted_at |
| feedbacks | AIMemory 환류 이력 | id, topic_card_id, target(txtaimemory), payload_summary, status, sent_at |
| settings | 앱 설정 | key, value_json, updated_at |

* `keyword_cache`/`embeddings`/`vec_index`/`clusters`는 언제든 재조회·재계산·재색인으로 복원 가능한 파생 데이터. 모델 교체 시 `embeddings`+`vec_index` 전체 재생성(마이그레이션).
* `topic_cards`가 유일한 "원본" 데이터 — 백업/내보내기 대상.
* 임베딩 벡터는 로컬에만 저장. 외부로 전송하지 않는다(소스측 공유(A)도 벡터만 수신, 클라우드로 벡터 미전송).

## 9. 시스템 아키텍처

| 영역 | 기술 | 비고 |
|-----|-----|-----|
| 프레임워크 | Tauri + React | 패밀리 스택 통일, TXTSpace와 컴포넌트 재사용 |
| 통합 조회 | HTTP 클라이언트(127.0.0.1) × 소스 수 + 공통 스키마 파서 | 소스별 동일 코드 재사용 |
| 임베딩 | 로컬 다국어 모델 **bge-m3**(1024-dim, Ollama/llama.cpp) + 증분 캐시 | 본문 미사용(로컬), 소스측 공유 시 벡터만 수신 |
| 벡터 스토어 | **sqlite-vec(vec0)** KNN, 규모 확장 시 HNSW | 앱 SQLite 내장, local-first |
| 발견 엔진 | Rust 코어 — 벡터 KNN·클러스터를 뼈대로 기간·빈도 융합 스코어(결정적) | 재현 가능·근거 산출 중심 |
| 합성(선택) | DeepSeek API 클라이언트(동의형) | 사용자 API Key, 기본 OFF |
| 저장 | SQLite(+ 필요 시 암호화) | topic_cards 원본 보호 |
| 환류 | TXTAIMemory MCP 클라이언트(쓰기, 동의형) | 미연결 시 비활성 |
| 딥링크 발신 | OS shell open으로 txtdiary:// / txtbrain:// / txtaimemory:// | 대상 미설치 시 안내 |

## 14. 접근성·프라이버시 원칙 요약 (패밀리 상속)

* 다크·고대비 기본, 색 이외 구분 병행, 최소 폰트 16px·타겟 50px.
