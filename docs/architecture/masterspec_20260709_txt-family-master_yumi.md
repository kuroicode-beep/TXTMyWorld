# TXT 패밀리 마스터 — 목적과 공통 프로토콜

맥락을 기록하고, 구조화하고, 연결해 나를 성장시키는 앱들의 집합.

문서 버전: v1.0 / 작성일: 2026-07-09 / 작성자: SVIL 유미 / 성격: 살아있는 규약 (수시 개정)

> 이 문서는 딱딱한 룰북이 아니다. TXT 패밀리 전체의 **존재 이유**와 앱들을 잇는 **공통 프로토콜**을 정의하되, 개별 앱이 성장하면서 규약도 함께 진화한다. 세부 규칙을 별도 문서로 못 박지 않고, 프로젝트/아이디어 문서를 중심으로 두며 이 문서는 계속 유연하게 갱신한다.

## 0. 한 줄 정의

TXT 패밀리는 **세 종류의 맥락(개인 기억·문서 지식·AI 대화)을 각각 맥락화**하고, 그것을 **하나의 지도로 보고(TXTSpace)**, **새로운 연결을 발견하는(TXTMyWorld, 가칭)** 개인 지능 생태계다.

## 1. 왜 존재하는가 (목적)

사람의 기억은 학교에서 있었던 일, 길에서 겪은 일, 집에서 생긴 일, 친구와의 일이 각각 따로 들어오지만, 잠자는 동안 뇌가 이것들을 묶고, 요약하고, 쓸모없는 건 버리고, 서로 다른 경험 사이에서 **새로운 연결**을 만든다. TXT 패밀리는 이 과정을 디지털로 재현한다.

* **문제:** 개인의 기록·지식·AI와의 작업 맥락이 세션마다, 앱마다, 파일마다 흩어진다. 다시 찾지 못하고, 연결되지 못하고, 성장으로 이어지지 못한다.
* **해결:** 맥락의 **종류별 전용 앱**으로 각각 잘 맥락화하고 → **공통 규격**으로 한 지도에 올리고 → **연결 레이어**에서 서로 다른 맥락 사이의 새 통찰을 캐낸다.
* **핵심 원리:** 단순 저장이 아니라 **재정리(consolidation)**. 쌓기만 하면 죽은 로그다. 묶고·요약하고·분류하고·망각해야 살아있는 기억이 된다.

## 2. 패밀리 구성 (맥락화의 3축 + 상위 2레이어)

| 계층  | 앱   | 맥락화 대상 | 핵심 질문 | 데이터 중심 | 상태  |
|-----|-----|--------|-------|--------|-----|
| 맥락화 | **TXTDiary** | 개인 기억·감정·경험 | 나는 무엇을 경험하고 느꼈는가? | DB (SQLite) | v1.0 RC 구현 완료 |
| 맥락화 | **TXTBrain** | 문서·지식 자료 | 이 생각과 연결되는 지식은? | DB 중심  | 개발 예정 |
| 맥락화 | **TXTAIMemory** | AI 대화·실작업 | AI와 무엇을 했고 무엇이 남았는가? | DB 중심  | PRD 작성 |
| 시각화 | **TXTSpace** | 위 3축의 통합 지도 | 내 맥락들은 어떻게 이어져 있는가? | read-only 소비 | RC 후 순차 개발 |
| 연결  | **TXTMyWorld** (가칭) | 3축의 연관성·새 연결 | 여기서 무엇이 새로 태어나는가? | 통합 조회  | 최종 목표 |

**보조 도구:**

* **TXTDrop** — 블록 설정 + 단축키로 텍스트를 즉시 캡처하는 입구. 현재는 txt 파일 저장 → TXTBrain 수동 수집 구조. TXTAIMemory와는 단축키로 대화 중 중요 텍스트를 **바로 원장에 꽂는** 빠른 캡처로 연계한다.
* **SAC API** — 문서 원본 저장을 담당하는 **별개 시스템**. TXT 패밀리의 DB 중심 앱들은 문서를 소유하지 않으며, 문서가 필요할 때 SAC API를 경유한다.

### 2.1 역할 경계 (중요)

* **TXTSpace = 보기.** 소스가 무엇이든 똑같이 지도에 얹는다. 데이터를 소유하거나 변경하지 않는다.
* **TXTMyWorld = 잇기·만들기.** 세 소스를 가로질러 연관을 찾고 새 맥락을 생성한다.
* **맥락화 3앱 = 각자의 원본 소유자.** 자기 도메인의 데이터를 소유하고, 공통 규격으로 read-only 제공한다.
* **문서 원본 ≠ 맥락 DB.** 문서 파일은 SAC API 소관. 맥락화 앱은 파일이 아니라 DB(키워드·연결·메타)를 다룬다.

## 3. 공통 프로토콜 — 연동 규격

TXT 패밀리를 하나로 묶는 진짜 뼈대. **맥락화 3앱(Diary/Brain/AIMemory)은 모두 동일한 read-only 연동 규격을 노출**한다. 그래야 TXTSpace는 소스 종류와 무관하게 지도에 얹고, TXTMyWorld는 소스를 가로질러 연결을 찾을 수 있다.

### 3.1 공통 Keyword/Context API 원칙

TXTDiary v1.0 RC에서 확정한 Keyword API schema v1.0을 패밀리 표준으로 승격한다.

* **localhost 전용:** 127.0.0.1에만 바인딩. 외부 네트워크 노출 금지.
* **read-only:** GET만 존재. POST/PUT/DELETE 없음(405).
* **페어링 토큰 인증:** 최초 연결 시 발급·사용자 승인. 토큰은 해시로만 저장.
* **본문 미포함:** 원본 텍스트(일기 본문, 문서 전문, 대화 전문)는 API 응답에 **절대 포함하지 않는다.** 키워드·빈도·감정·동시출현 등 메타데이터만 제공.
* **schema_version 필수:** 모든 응답에 버전 필드. 패밀리 전체가 동기화한다.
* **기본 OFF:** 사용자 동의 후 활성화.

### 3.2 공통 응답 스키마 (schema v1.0 상속)

```json
{
  "schema_version": "1.0",
  "source": "txtdiary | txtbrain | txtaimemory",
  "generated_at": "ISO8601",
  "date_range": {"from": "date", "to": "date"},
  "keywords": [
    {
      "text": "string",
      "normalized_text": "string",
      "category": "topic|emotion|person|work|creative|other",
      "frequency": 0,
      "avg_emotion_score": 0.0,
      "first_seen": "date",
      "last_seen": "date",
      "cooccurrence": [{"text": "string", "count": 0}]
    }
  ]
}
```

* `source` 필드로 어느 맥락에서 왔는지 구분한다. TXTSpace/TXTMyWorld는 이 필드로 노드 색·레이어를 나눈다.
* 각 앱은 자기 도메인에 필요한 필드를 **확장**할 수 있으나(예: TXTAIMemory의 `ai_id`), 위 코어 필드는 공통 유지한다.
* 확장 필드는 schema_version 마이너 증가로 관리하고, 소비 측은 모르는 필드를 무시(graceful)한다.

### 3.3 공통 엔드포인트 최소 집합

| Endpoint | Method | 목적  |
|----------|--------|-----|
| /health  | GET    | 상태, schema_version, source, 앱 버전 |
| /keywords | GET    | 키워드 목록. from/to/category/min_frequency/limit |
| /keywords/{normalized_text}/cooccurrence | GET    | 동시 출현 통계 |

### 3.4 딥링크 규약

각 앱은 자기 스킴을 등록해, TXTSpace/TXTMyWorld 노드 클릭 시 해당 앱의 상세로 돌아올 수 있게 한다.

* TXTDiary: `txtdiary://search?keyword={keyword}`
* TXTBrain: `txtbrain://search?keyword={keyword}` (예정)
* TXTAIMemory: `txtaimemory://recall?keyword={keyword}` (예정)

## 4. AI 공통 규칙 (살아있는 섹션)

> 이 섹션은 고정 룰북이 아니라 계속 진화한다. TXTAIMemory가 AI 대화·작업 맥락을 다루는 앱인 만큼, AI 관련 공통 규칙은 자연스럽게 여기(그리고 TXTAIMemory PRD)에서 자란다.

* **재정리 우선:** AI가 맥락을 다룰 때 목표는 저장이 아니라 재정리다. 원장(raw) → 통합(consolidated) → 망각(decay) 루프를 기본 사고틀로 삼는다.
* **본문 최소주의:** 외부/상위 레이어로 나가는 데이터는 메타데이터 우선. 원문 전문은 소유 앱 안에 머문다.
* **AI 구분 태깅:** 여러 AI 에이전트(유미/소미/루미/다미/타미/세미 등)의 맥락은 `ai_id`로 구분 가능해야 하고, 동시에 태그 없이 통합 조회도 가능해야 한다. (사람의 "학교 일만 떠올리기" vs "인생 전체 돌아보기")
* **망각은 기능이다:** 크기를 일정 선에서 유지하기 위해, 통합이 끝난 원장과 오래 안 불린 기억은 삭제 가능해야 한다. 개별 맥락 데이터의 수동 삭제도 허용한다.
* **진단 금지:** 감정·상태 분석은 진단이 아니라 경향 제안이다. (TXTDiary 정서 안전장치 원칙 상속)
* **동의 기반 외부 전송:** 민감한 원문은 사용자가 명시적으로 허용한 경우에만 외부 API로 나간다.

## 5. 개발 순서 (의존성)

1. **TXTDiary v1.0 RC** — 구현 완료. Keyword API schema v1.0의 원본 정의처.
2. **TXTAIMemory** — PRD 작성. DB 중심 + consolidation 루프 + TXTDrop 연계.
3. **TXTBrain** — 문서 지식 맥락화. 공통 규격 상속.
4. **TXTSpace** — 3축이 규격을 갖추면 지도로 통합.
5. **TXTMyWorld** — 최종. 3축을 가로지르는 연결·생성 레이어.
